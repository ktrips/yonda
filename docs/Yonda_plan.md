## この 1 枚だけ読めばいい（Jobs 型圧縮）

**残す 1 メッセージ: 「読んだ本が、ぜんぶ自分の財産になる。」**
（顧客の痛み「最近何読んだ？に答えられない」への“答え”を、能力ではなく得られる価値で言う）

- **誰に**: Audible・Kindle・図書館を併用していて、読んだ本を手入力で記録する気はないが「読了が勝手に貯まる」なら欲しい読書家（Thiel）
- **何を**: 手入力ゼロの読書記録。同期も書評も AI がやる。あなたは読むだけ（Bezos）
- **どう検証**: 次の 48 時間でトップ文言を 1 メッセージに差し替え、訪問→ログイン率 5% を測る（Musk）
- **どう回す**: アプリ → 開発ガイド本（KDP）→ 読者 → ユーザーの複利ループ（Sam）
- **何を言わないか**: 対応サービス一覧・AI 搭載・機能紹介。1 メッセージを邪魔するものは全部後ろへ（Jobs）

## Part 7: Audible × Kindle Unlimited アフィリエイト導線設計

### 7-1. なぜ今これが P0 なのか

| 指標 | 現状 | 目標 |
|---|---|---|
| 既存アフィリエイト | 検索ボタンに `tag=ktrip-22` を付与済み。クリックされても「検索結果を眺めて閉じる」がほとんど | — |
| Audible 無料体験 | 30日間無料（以降 月1,500円）。1件成立で **1,500円前後**の紹介料 | 月5件 = 月7,500円 |
| Kindle Unlimited 無料体験 | 30日間無料（以降 月980円）。1件成立で **数百円** の紹介料 | 月10件 = 月数千円 |
| 現在の導線 | **ゼロ**。Audible/KU の無料体験ページへの誘導が一切ない | 3箇所に CTA を設置 |

Yonda のユーザーは「Audible・Kindle を使っている読書家」がコアターゲット。  
つまり **Audible/KU を「まだ使っていない友人・家族に紹介したくなる」最も動機が高い人たちがすでにアプリを使っている**。ここに導線がないのは損失。

### 7-2. 使用する URL（アソシエイトタグ込み）

```
# Audible 30日無料体験（LP）
https://www.amazon.co.jp/dp/B00NB86OYE?tag=ktrip-22

# Kindle Unlimited 30日無料体験（LP）
https://www.amazon.co.jp/kindle-dbs/hz/subscribe/ku?tag=ktrip-22

# Audible 検索（既存）
https://www.audible.co.jp/search?keywords={q}&tag=ktrip-22

# Kindle 検索（既存）
https://www.amazon.co.jp/s?k={q}&i=digital-text&tag=ktrip-22
```

### 7-3. 導線の設置場所（✅ 4 箇所すべて実装済み — 2026-07-10）

#### ① ランディングページ（未ログイン）【✅ 実装済み】

未ログイン訪問者への最大露出面。メイン CTA「Googleで無料ではじめる」の下に二次 CTA。

```
[ 読んだ本が、あなたの財産に。]
 [Googleで無料ではじめる]
 すでに ◯ 冊の本が登録されています
 ── AudibleやKindle Unlimitedをまだ試していない方へ ──
 [🎧 Audible 30日無料体験]  [📖 Kindle Unlimited 30日無料]
```

#### ② Yomu ページ（AI 推薦書の下）【✅ 実装済み】

「次に読む本」を探している瞬間 = 購入意欲が最も高い。各推薦書カードの下にボタン。

```
[ 推薦書カード ]
  タイトル・著者・AI コメント
  Amazon / Audible / メルカリ / ブックオフ / 図書館
  [🎧 Audibleで聴く]  [📖 KUで読む]
```

#### ③ 書籍詳細モーダル（図書館・紙の本のみ）【✅ 実装済み】

図書館・紙の本の詳細を見ている時だけ表示。**Audible/Kindle ソースには非表示**（保持済みのため）。

#### ④ ハンバーガーメニュー底部【✅ 実装済み】

毎セッションで目に入る場所。読了記録を眺めた後 = 満足感が高い瞬間に露出。

```
 📚 まだ使っていない方へ
 [🎧 Audible 30日無料体験]
 [📖 Kindle Unlimited 30日無料体験]
 ※アフィリエイトリンクです
```

### 7-4. 景表法・ステマ規制への対応（✅ 実装済み — 2026-07-10）

| 項目 | 対応 | 状況 |
|---|---|---|
| アフィリエイト明示 | 各 CTA 近くに「※アフィリエイトリンクです」を表示 | ✅ |
| フッター表記 | 「当サービスはAmazonアソシエイト・プログラムの参加者です。適格販売により収入を得ることがあります。」を全ページに表示 | ✅ |
| rel 属性 | 全 CTA に `rel="noopener sponsored"`（Google ステマ規制ガイドライン対応） | ✅ |
| 誇張表現の禁止 | 「必ずお得」「全員無料」などの断定表現は使わない | ✅ |

### 7-5. 成果計測（✅ 実装済み — 2026-07-10）

| **Sam 型（Part 3）** | どう作り続けるか | アプリ・書籍・アダプタ・デプロイ基盤を資産化し複利で回す |
| **Musk 型（Part 4）** | どの前提を壊すか | ストア配布・アプリ内課金の前提を捨て、Web 直配布で実験速度 10 倍 |
| **Jobs 型（Part 5）** | 何を削るか | 「読んだ本が、ぜんぶ自分の財産になる」に圧縮し、機能紹介を最初の 10 秒から消す |

# Part 1: 独占できる小さな市場（Thiel 型）

「読書記録アプリ」の大きな市場は多重に赤い海:

| 領域 | 支配者 | 戦うと何が起きるか |
|---|---|---|
| 読書 SNS | 読書メーター・ブクログ | コミュニティの厚みに勝てない |
| 蔵書管理 | ビブリア等の無数のアプリ | 手入力前提の機能差で価格競争 |
| 海外 | Goodreads（Amazon 傘下） | 英語圏・Amazon 統合で正面衝突 |

軸で分解すると、競合は「手入力で記録して仲間と共有したい人」を取り合っている。**「Audible・図書館・Kindle を併用し、手入力はしたくないが読了は貯めたい人」** がガラ空き。

勝てる理由 3 つ:

1. **技術**: Audible 非公式 API・図書館スクレイピング・Kindle 連携という「面倒な実装」自体が参入障壁。大手には市場が小さすぎる
2. **流通**: 開発ガイド書籍（KDP）= 「Cursor で個人開発したい人」だけが検索して届く自前チャネル。広告費ゼロ
3. **構造**: 固定費が月数百円なので、ニッチでも黒字化できる（大手の ROI に合わない = 攻めてこない）

### 最初のオファー文

> **Audible も図書館も Kindle も使うあなたへ。**
>
> 読んだ本、思い出せますか。記録アプリは、どうせ続かないことをあなたが一番よく知っているはずです。
>
> Yonda は Audible・Kindle・図書館の記録を毎朝勝手に同期します。あなたが入力する項目はありません。読むだけです。
>
> 読んだ本が、ぜんぶ自分の財産になります。参加は無料です。

# Part 2: 顧客から逆算する PR/FAQ（Bezos 型）

## 2-1. 顧客の一番強い痛み

**「読んでいるのに、何を読んだか答えられない」**

想定顧客を 1 人に絞る:

> 40 代・通勤で Audible、寝る前に Kindle、子どもと図書館。月 5〜6 冊は「読んでいる」のに、
> 「最近何読んだ？」と聞かれると 2 冊しか思い出せない。読書メーターは 3 日で放置した。
> 原因は記録が面倒だからで、読書量ではない。

売るものは記録機能ではなく **「読んできた自分が見える」こと**。

## 2-2. 未来のプレスリリース（2026 年 12 月想定）

> ### 手入力ゼロの読書記録 Yonda、読了自動記録の継続率 ◯% を達成
> ### ― 記録が 3 日で終わっていた読書家が、1 年分の読了リストを持てるようになった ―
>
> Yonda は Audible・Kindle・公共図書館・紙の本の記録を毎朝自動同期する。ユーザーが入力する項目は原則ゼロ。読了した本は AI が書評ポイントを生成し、次に読む本まで提案する。

（◯% を言えるようにする計測が P0-2 / P1 の実装要件）

## 2-3. FAQ（購入前に顧客が聞く 10 個 — P1 で確定）

骨子: 無料で何ができるか / 対応サービス / 自分の図書館は使えるか / 認証情報は安全か / データはエクスポートできるか / 月額の元は取れるか / 解約は簡単か — **この答えがそのまま無料枠・価格・プライバシー方針の仕様になる。**

## 2-4. 今すぐ直すべき商品設計（顧客から逆算した指摘）

1. **「AI 機能は自分の API キーを設定」を廃止する（P2 最優先）** — 想定顧客は API キーという概念を知らない。サーバー代理呼び出しにして原価を Plus に織り込む
2. **登録率・継続率を計測する仕組みがない（P0）** — 未来 PR の数字は計測があって初めて言える
3. **トップの最初の 10 秒が機能説明になっている（P0）** — 1 メッセージ + 最初の行動（Google ログイン）に差し替える

# Part 3: 資産化・仕組み化プラン（Sam 型）

| 資産 | 内容 | 状態 |
|---|---|---|
| A1 アプリ本体 | yonda.ktrips.net | ✅ 稼働中 |
| A2 書籍パイプライン | md → Kindle 判型 docx（docs/build_docx.py）。2 冊目からの限界コストほぼゼロ | ✅ 実装済み / 🚧 1 冊目の入稿待ち |
| A3 汎用図書館アダプタ | 設定を足せば別図書館に対応 | ✅ 設計済み |
| A4 デプロイ・運用テンプレ | deploy.sh / .gcloudignore / Cloud Scheduler 構成 | ✅ 完成 |
| A5 開発ガイド（公開） | /dev-guide — それ自体が SEO 面・書籍の Web 版 | ✅ 公開中 |

**循環**: アプリを作る → 開発記を本にする → 本の読者がユーザーになる → 利用データと改善が次の章・次のアプリを生む。

判断基準: **2 回やった作業はテンプレ化してから終える。**

# Part 4: 第一原理で 10 分の 1 にする（Musk 型）

| 前提 | 正体 | 判定 |
|---|---|---|
| 「アプリはストアで配布」 | 慣習。Web なら審査ゼロ、1 日に何度でもデプロイできる | **消した**（✅ 現構成） |
| 「収益化はアプリ内課金」 | 慣習。Web なら手数料 15〜30% なしで Stripe・アフィリエイト・書籍導線 | **消した**（✅/🚧） |
| 「機能を増やせば価値が上がる」 | 思い込み。検証すべきは 1 メッセージだけ | **消す**（P0） |
| ユーザー認証情報・個人データ保護 | 法的・信頼の制約 | **残す** |

律速は「仮説を顧客に届けて検証する速度」— 技術面は解決済み（デプロイ 1 分）。**未解決なのは計測**。P0-2 が全実験の前提になる。

失敗した場合の学び方（層の特定）:

| 結果 | 外れた層 | 次の一手 |
|---|---|---|
| 訪問はあるが登録率 < 2% | 訴求（言葉） | 見出しを痛みの承認型に差し替え 24h 再テスト |
| 訪問自体が集まらない | チャネル | 書籍/ガイド経由と SNS 経由を分計し、効く方に集中 |
| 登録するが同期設定まで進まない | 初期体験 | 設定完了までの動線を 1 画面に削る |
| 使うが課金しない（P2 以降） | 価格・特典 | Plus 特典を「原価がかかるもの」に寄せ直す |

# Part 5: 削って 1 つの体験にする（Jobs 型）

**顧客が覚えるべき 1 文: 「読んだ本が、ぜんぶ自分の財産になる。」**
（「3秒で答えられる」は“できること”で、顧客の希望＝“得られる価値”になっていなかったため改訂）

捨てる要素（最初の 10 秒から消す。機能は消さない）:

| 捨てる要素 | 削った理由 |
|---|---|
| 対応サービスのロゴ・機能一覧の先出し | 「多機能=面倒そう」と記録アプリ放置の記憶を呼び起こす |
| 「AI 搭載」の強調 | 顧客は AI ではなく「思い出せる自分」が欲しい。2026 年に差別化にもならない |
| ランキング・コミュニティの先出し | 想定顧客は共有したい人ではない。ログイン後に出会えばいい |

伝える順番: ① 痛みの承認（「読んだ本、思い出せない」）→ ② 約束（1 文）→ ③ 根拠 1 つ（自動同期・入力ゼロ）→ ④ 最初の行動（Google ログイン）→ ⑤ 証拠（みんなの読了数）→ ⑥ 価格（無料で始まることだけ）。

# Part 6: 統合ロードマップ

**小さな市場（Thiel: 併用読書家）に「読んだ本が、ぜんぶ自分の財産になる」だけを伝え（Jobs）、その約束を顧客の変化から設計し（Bezos: PR/FAQ）、Web 直配布 + 計測の最速サイクルで検証し（Musk）、アプリ→書籍→読者→ユーザーの資産化ループ（Sam）で回す。**

## 判断基準（迷ったらこれ）

1. **競争していないか** — 読書メーターの土俵に乗っていないか。乗っているならニッチの言葉に言い換えられるまで削る（Thiel）
2. **顧客の変化を言えるか** — 「併用読書家の何が変わるか」を説明できない機能は作らない（Bezos）
3. **2 回目があるか** — 2 回やりそうな作業はテンプレ化してから終える（Sam）
4. **その工程は物理法則か慣習か** — 48 時間で数値検証できない施策は、検証できる形に分解してから着手（Musk）
5. **1 メッセージを強めるか** — 薄めるなら最初の 10 秒から外し、ログイン後の奥に置く（Jobs）
